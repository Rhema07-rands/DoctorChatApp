1.  PROJECT OVERVIEW & ARCHITECTURE
================================================================================

DoctorChatApp is a full-stack telemedicine mobile application built with:

  FRONTEND (Mobile App):
    - React Native + Expo (managed workflow with custom dev client)
    - Expo Router (file-based navigation)
    - TypeScript
    - @microsoft/signalr (real-time messaging)
    - react-native-webrtc (video/audio calls)

  BACKEND (API Server):
    - ASP.NET Core 8 Minimal API (single Program.cs)
    - Entity Framework Core with MySQL (database ORM)
    - SignalR Hubs (real-time communication)
    - JWT Authentication (token-based security)
    - Cloudinary (image/file upload storage)
    - BCrypt (password hashing)

  HOW THEY CONNECT:
    Mobile App  ──HTTP REST──>  ASP.NET API  ──EF Core──>  MySQL Database
    Mobile App  ──SignalR WS──> SignalR Hubs  (real-time chat + call signaling)
    Mobile App  ──WebRTC P2P──> Other Mobile App  (direct audio/video streams)

  USER ROLES:
    1. Doctor  - Manages appointments, consultations, chats, availability
    2. Patient - Browses doctors, books appointments, chats, makes calls

  NAVIGATION STRUCTURE:
    / (index.tsx)          -> Splash screen / auto-login check
    /login                 -> Login form
    /register              -> Registration form (doctor or patient)
    /(tab)/Doctor_page/*   -> Doctor's 5 main tabs (Home, Chats, Schedule, Consults, Account)
    /(tab)/Patient_page/*  -> Patient's 5 main tabs (Home, Chats, Doctors, Rx, Account)
    /Doctor_subpage/*      -> Doctor's detail screens (chat, calls, settings)
    /Patient_subpage/*     -> Patient's detail screens (chat, calls, booking, doctor profile)

--- API ENDPOINTS ---

  POST /api/auth/register
    -> Creates a new user account
    -> Hashes password with BCrypt.HashPassword()
    -> Returns JWT token + user profile

  POST /api/auth/login
    -> Validates email/password
    -> Uses BCrypt.Verify() to check password against hash
    -> Generates JWT token with user claims (Id, Email, Role)
    -> Returns token + user profile

  GET /api/doctors/search?query=...
    -> Searches doctors by name, specialization, or conditions
    -> Returns list of matching doctors with their profiles

  GET /api/appointments
    -> Returns all appointments for the authenticated user
    -> Filters by doctor or patient based on user role

  POST /api/appointments
    -> Creates a new appointment
    -> Sends real-time notification to the doctor via SignalR

  PUT /api/appointments/{id}/status
    -> Updates appointment status (Confirm, Complete, Cancel)
    -> Notifies both parties via SignalR

  PUT /api/appointments/{id}/reschedule
    -> Changes appointment date/time
    -> Resets status to "Pending" for re-confirmation

  GET /api/chat/{userId}
    -> Returns chat history between current user and specified user
    -> Marks messages as read

  POST /api/upload
    -> Uploads a file to Cloudinary
    -> Returns the CDN URL

  POST /api/calls/initiate
    -> Creates a call record in the database
    -> Sends "IncomingCall" signal to receiver via SignalR

  PUT /api/calls/{id}/answer
    -> Marks call as answered
    -> Sends "CallAnswered" signal to caller

  PUT /api/calls/{id}/end
    -> Ends the call, records end time

  GET /api/calls/turn-credentials
    -> Returns TURN server credentials for WebRTC NAT traversal

  POST /api/reviews
    -> Adds a patient review for a doctor

  PUT /api/reviews/{id}
    -> Updates an existing review (only by the review author)

  DELETE /api/reviews/{id}
    -> Deletes a review (only by the review author)

  POST /api/notifications/register-token
    -> Registers device push notification token

  PUT /api/profile/update
    -> Updates user profile information

  PUT /api/profile/picture
    -> Updates profile picture URL


--- SIGNALR HUBS ---

  ChatHub (/chathub):
    -> Handles real-time messaging
    -> SendMessage() -> Saves message to DB, sends to receiver, confirms to sender
    -> Joins users to groups based on their user ID for targeted messaging

  SignalingHub (/signalinghub):
    -> Handles WebRTC call signaling
    -> CallUser()      -> Sends IncomingCall to receiver
    -> AnswerCall()    -> Sends CallAnswered to caller
    -> EndCall()       -> Sends CallEnded to other party
    -> SendOffer()     -> Forwards WebRTC SDP offer
    -> SendAnswer()    -> Forwards WebRTC SDP answer
    -> SendCandidate() -> Forwards ICE candidates

--- src/config.ts ---
PURPOSE: Centralizes the API base URL so every service uses the same address.

WHY THIS EXISTS: Without this, you'd have to hardcode the IP in every file,
and change it whenever your network changes. This auto-detects it.


--- src/services/authEvents.ts ---
PURPOSE: A simple event emitter for authentication events (specifically 401 errors).

  Lines 1-17: AuthEvents class
    -> Maintains a list of callback functions (listeners)
    -> onUnauthorized(callback): Registers a listener, returns unsubscribe function
    -> emitUnauthorized(): Calls ALL registered listeners

  Line 19: export const authEvents = new AuthEvents();
    -> Creates a singleton instance used across the app

WHY THIS EXISTS: When ANY API call returns 401 (token expired), the api.ts
interceptor calls authEvents.emitUnauthorized(). UserContext listens for this
and automatically logs the user out. This is a DECOUPLED approach — the API
service doesn't need to know about navigation or user state. It just fires
an event, and whoever is listening handles it.


--- src/services/api.ts ---
PURPOSE: Creates a configured Axios HTTP client that ALL API calls use.

  Lines 7-12: axios.create()
    -> Creates an axios instance with baseURL pointing to the backend
    -> Default Content-Type is JSON

  Lines 15-28: REQUEST INTERCEPTOR
    -> Runs BEFORE every HTTP request is sent
    -> Reads the JWT token from SecureStore (encrypted storage)
    -> Attaches it as "Authorization: Bearer <token>" header
    -> This means you never have to manually add auth headers to API calls

  Lines 31-42: RESPONSE INTERCEPTOR
    -> Runs AFTER every HTTP response is received
    -> If response status is 401 (Unauthorized):
       -> Emits the 'unauthorized' event via authEvents
       -> UserContext will catch this and log the user out
    -> All other errors are passed through normally

WHY THIS EXISTS: Centralizes auth header injection and token expiry handling.
Every file that imports { api } gets automatic authentication.

  Lines 6-21: CALLBACK ARRAYS

--- src/services/webRtcService.ts ---
PURPOSE: Manages WebRTC peer-to-peer connections for video/audio calls.

  WebRTC (Web Real-Time Communication) enables direct audio/video streaming
  between two devices without going through the server.

  Lines 11-24: ICE SERVER CONFIGURATION
    -> STUN servers help devices discover their public IP address
    -> Google provides free STUN servers
    -> TURN servers relay traffic when direct connection fails (firewalls, NAT)
    -> iceTransportPolicy: 'all' means try direct first, then TURN relay

  Lines 26-36: CLASS PROPERTIES
    -> peerConnection: The WebRTC connection object
    -> localStream: Your camera/mic stream
    -> remoteStream: The other person's camera/mic stream
    -> pendingCandidates: ICE candidates received before connection is ready
    -> pendingOffer: SDP offer received before peer connection is created

  Lines 42-74: setupMedia()
    -> Enumerates available cameras, picks front-facing by default
    -> Requests camera AND microphone access from the OS
    -> Audio settings: echoCancellation, noiseSuppression, autoGainControl
    -> Video settings: 640x480 at 30fps, front-facing camera
    -> For audio-only calls, video is set to false
    -> Returns the MediaStream object

  Lines 76-101: refreshIceServers()
    -> Fetches TURN server credentials from the backend
    -> TURN credentials expire, so we fetch fresh ones before each call
    -> Prioritizes TURN servers (for reliability) over STUN servers
    -> Falls back to Google STUN servers if fetch fails

  Lines 103-173: createPeerConnection()
    -> Creates the RTCPeerConnection — the core WebRTC object
    -> Adds local media tracks (your camera/mic) to the connection
    -> Sets up event listeners:
       - 'track': When remote media arrives, creates/updates remoteStream
       - 'icecandidate': Sends ICE candidates to the other peer via SignalR
       - 'iceconnectionstatechange': Logs connection progress
       - 'connectionstatechange': When connected, flushes queued candidates
    -> Processes any buffered offer (received before PC was ready)

  Lines 175-193: createOffer()
    -> Caller creates an SDP offer (describes their media capabilities)
    -> Sets it as local description
    -> Sends it to the other peer via SignalR

  Lines 195-222: handleOffer()
    -> Receiver processes the caller's SDP offer
    -> Sets it as remote description
    -> Flushes queued ICE candidates
    -> Creates an SDP answer (describes receiver's media capabilities)
    -> Sends answer back via SignalR

  Lines 224-236: handleAnswer()
    -> Caller processes the receiver's SDP answer
    -> Sets it as remote description
    -> Flushes queued ICE candidates

  Lines 238-266: handleCandidate() & processPendingCandidates()
    -> ICE candidates describe network paths between the two peers
    -> If the connection isn't ready yet, candidates are queued
    -> Once ready, all queued candidates are flushed and applied

  Lines 291-336: closeConnection()
    -> Stops ALL local media tracks (releases camera/mic hardware)
    -> Stops ALL remote media tracks
    -> Closes the peer connection
    -> Resets all state
    -> This is critical — without proper cleanup, the camera stays on!

  Lines 337-361: CALL CONTROLS
    -> toggleMute(): Enables/disables audio tracks
    -> toggleVideo(): Enables/disables video tracks
    -> switchCamera(): Switches between front and back camera

      1. Caller calls startCall():
         - Sets up local media (camera/mic)
         - Starts InCallManager for audio routing
         - Initiates call via REST API (/calls/initiate)
         - Creates peer connection
         - Navigates to call screen

      2. Receiver gets IncomingCall signal:
         - Plays ringtone
         - Shows incoming call screen
         - User taps Accept or Decline

      3. If accepted (acceptCall()):
         - Stops ringtone
         - Sets up local media
         - Creates peer connection
         - Answers via REST API (/calls/{id}/answer)

      4. Caller gets CallAnswered with isAccepted=true:
         - Creates WebRTC offer
         - Sends to receiver via SignalR

      5. Receiver gets ReceiveOffer:
         - Sets remote description
         - Creates answer
         - Sends back via SignalR

      6. Both sides exchange ICE candidates
         -> Direct connection established
         -> Audio/Video streams flow peer-to-peer

      7. Either party calls endCall():
         - Stops media tracks
         - Closes WebRTC connection
         - Notifies server
         - Resets state

    Audio Device Management:
      -> Listens for onAudioDeviceChanged events
      -> Auto-switches to Bluetooth when it becomes available
      -> Falls back to earpiece/speaker when Bluetooth disconnects

13. GLOSSARY OF KEY CONCEPTS
================================================================================

JWT (JSON Web Token):
  A compact, self-contained token used for authentication. Contains the user's
  ID, email, role, and expiry time. The backend creates it on login, the app
  stores it securely, and sends it with every API request as proof of identity.

SignalR:
  Microsoft's library for real-time web communication. Uses WebSockets when
  available, falls back to other transports. In this app, it enables:
  - Instant message delivery
  - Live call signaling
  - Real-time availability updates
  - Appointment notifications

WebRTC (Web Real-Time Communication):
  A protocol for direct peer-to-peer audio/video communication. The connection
  setup (signaling) goes through SignalR, but once connected, media streams
  flow directly between the two devices without going through the server.

SDP (Session Description Protocol):
  Describes the media capabilities of a WebRTC peer (codecs, formats, etc.).
  The "offer" describes what the caller can send/receive. The "answer" describes
  what the receiver can handle. They negotiate compatible media formats.

ICE (Interactive Connectivity Establishment):
  The process of finding the best network path between two peers. ICE candidates
  represent possible network routes. STUN servers help discover public IPs.
  TURN servers relay traffic when direct connection is impossible.

React Context:
  React's built-in state management. Providers wrap the component tree and make
  state available to all descendants without prop drilling. In this app, we use
  6 contexts: User, Theme, Appointments, Calls, Notifications, DoctorDirectory.

Expo Router:
  File-based routing for React Native. The file structure in app/ maps directly
  to URL routes. (tab)/ creates tab navigation. _layout.tsx files define
  navigation containers. Files like login.tsx become /login routes.

SecureStore:
  Expo's encrypted storage (Keychain on iOS, EncryptedSharedPreferences on Android).
  Used to store sensitive data like JWT tokens that persist across app restarts.

BCrypt:
  A password hashing algorithm. Converts passwords into irreversible hashes.
  Even if the database is compromised, passwords can't be recovered.
  BCrypt.Verify() checks if a plain password matches a stored hash.

Entity Framework Core:
  Microsoft's ORM (Object-Relational Mapper) for .NET. Lets you write C# code
  instead of SQL queries. DbContext is the main class, DbSet<T> represents
  database tables, and LINQ queries are translated to SQL automatically.

Cloudinary:
  A cloud service for image and file storage. Files are uploaded to Cloudinary's
  CDN and accessed via URLs. Used for profile pictures and chat attachments.

1. Set up the backend first (ASP.NET + MySQL + SignalR)
2. Create the React Native project with Expo
3. Build the service layer (API, Auth, SignalR, WebRTC)
4. Create the context providers (User, Theme, Appointments, etc.)
5. Build the screens one by one, starting with auth screens
6. Add real-time features (chat, calls) last
